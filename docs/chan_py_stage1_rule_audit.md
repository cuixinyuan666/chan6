# chan.py stage-1 rule audit

This document distills local hichan source snippets into Rust alignment rules.

The full local snippet file is generated by:

```bash
HICHAN_REPO="$HOME/Downloads/chan_replay_app_v0_1/chan_replay_app" \
python tools/extract_hichan_stage1_snippets.py \
  --out docs/chan_py_stage1_source_snippets.md
```

Do not commit the generated snippet file unless it is intentionally reviewed for source-size and path safety. This audit document records only the rules required for Chan6 Rust implementation.

## 1. Include / merged K-line rules

hichan source:

```text
Combiner/KLine_Combiner.py
KLine/KLine_List.py
```

Observed rules:

```text
1. KLine_List.add_single_klu appends the first CKLine directly.
2. For later raw K-lines, it calls last CKLine.try_add(klu).
3. CKLine_Combiner.test_combine classifies the relationship as COMBINE / UP / DOWN / INCLUDED.
4. If output is COMBINE, the new raw K-line is appended into the current combined K-line.
5. If output is not COMBINE, a new CKLine is appended and the middle CKLine updates FX using previous/current/next.
```

The important include update rule:

```text
When combined direction is UP:
  high = max(previous high, new high)
  low  = max(previous low, new low)

When combined direction is DOWN:
  high = min(previous high, new high)
  low  = min(previous low, new low)
```

Chan6 current status:

```text
crates/chan6_core/src/chan/include.rs already follows the UP/DOWN max-max / min-min rule.
```

Known gaps:

```text
1. hichan has allow_top_equal handling.
2. hichan has exclude_included handling.
3. hichan has special handling for one-price K-line cases.
4. hichan get_peak_klu scans from the end of the internal list, so equal high/low anchors may prefer the later raw K-line.
```

Next fixtures needed:

```text
1. equal-high include fixture
2. equal-low include fixture
3. one-price K-line include fixture
4. expanding containing-bar fixture with equal boundary
```

## 2. FX rules

hichan source:

```text
Combiner/KLine_Combiner.py
KLine/KLine.py
KLine/KLine_List.py
Common/CEnum.py
```

Observed default FX rule from CKLine_Combiner.update_fx:

```text
TOP:
  previous.high < current.high
  next.high     < current.high
  previous.low  < current.low
  next.low      < current.low

BOTTOM:
  previous.high > current.high
  next.high     > current.high
  previous.low  > current.low
  next.low      > current.low
```

Chan6 current status:

```text
crates/chan6_core/src/chan/fx.rs uses strict three-merged-bar FX detection matching this default strict inequality pattern.
```

Known gaps:

```text
1. hichan has exclude_included FX mode.
2. hichan has allow_top_equal behavior.
3. FX validity for BI is not equal to raw FX detection; BI also calls CKLine.check_fx_valid.
```

Next fixtures needed:

```text
1. equal-top non-FX fixture
2. equal-bottom non-FX fixture
3. exclude_included / allow_top_equal fixture when config exposes it
```

## 3. BI rules

hichan source:

```text
Bi/Bi.py
Bi/BiConfig.py
Bi/BiList.py
KLine/KLine.py
Common/CEnum.py
Common/func_util.py
```

Observed BI creation pipeline:

```text
1. KLine_List.add_single_klu calls bi_list.update_bi(middle_klc, last_klc, step_calculation).
2. CBiList.update_bi calls update_bi_sure.
3. Before the first BI exists, CBiList uses free_klc_lst to cache FX K-lines.
4. A first BI is created only when two opposite FX K-lines pass can_make_bi.
5. add_new_bi creates CBi(pre_klc, cur_klc, idx, is_sure).
```

Observed can_make_bi gates:

```text
can_make_bi = satisfy_bi_span && check_fx_valid && end_is_peak
```

Observed strict span rule:

```text
if bi_algo == 'fx':
  skip span check
else if is_strict:
  require klc.idx - last_end.idx >= 4
else:
  require weaker span / raw unit count rule
```

Observed endpoint value rules from CBi:

```text
UP BI:
  begin value = begin_klc.low
  end value   = end_klc.high
  begin raw anchor = begin_klc low peak KLU
  end raw anchor   = end_klc high peak KLU

DOWN BI:
  begin value = begin_klc.high
  end value   = end_klc.low
  begin raw anchor = begin_klc high peak KLU
  end raw anchor   = end_klc low peak KLU
```

Observed end_is_peak rule:

```text
If last_end is BOTTOM:
  between last_end.next and cur_end, no intermediate high may exceed cur_end.high.

If last_end is TOP:
  between last_end.next and cur_end, no intermediate low may go below cur_end.low.
```

Chan6 current status:

```text
1. bi.rs now requires default merged-index span >= 4.
2. Stage-1 gold fixtures cover:
   - adjacent top/bottom does not form BI
   - far enough top/bottom forms one down BI
3. BI fields are tested against hichan gold for start/end raw index, price, direction, and is_sure.
```

Known gaps:

```text
1. Chan6 does not yet implement CKLine.check_fx_valid gates.
2. Chan6 does not yet implement end_is_peak gates.
3. Chan6 does not yet model bi_algo='fx'.
4. Chan6 does not yet model non-strict BI span behavior.
5. Chan6 does not yet model virtual BI / step_calculation behavior.
6. Chan6 does not yet model BI endpoint update/replacement behavior.
```

Next fixtures needed:

```text
1. check_fx_valid failure fixture
2. end_is_peak failure fixture
3. bi_algo='fx' fixture
4. non-strict BI fixture
5. BI endpoint update fixture
6. step/virtual BI fixture
```
